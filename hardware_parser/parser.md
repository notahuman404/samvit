# HWDL — Hardware Description Language Specification
## Version 1.0 | Compiler Design & Language Reference

---

## 1. Language Goals

### What HWDL Is

HWDL (Hardware Description Language) is a **strict, declarative DSL** for describing electronic hardware at the logical schematic level. It captures:

- Component definitions (parts) and their pin interfaces
- Power domains and their electrical properties
- Named net connectivity between component pins
- Modules (reusable subsystems) containing instances and connections
- Constraints, placement hints, routing hints, and simulation/validation metadata
- Compile targets that downstream tooling (schematic, PCB, netlist generators) will consume

HWDL is designed to be generated by AI agents and parsed by deterministic compiler tooling. Every construct has a single, unambiguous interpretation.

### What HWDL Is Not

- Not an RTL/HDL language (no Verilog/VHDL behavioral semantics)
- Not a netlist format (HWDL compiles *to* a structured IR from which netlists are generated)
- Not a PCB layout tool (placement/routing hints are advisory only)
- Not a simulation language (simulation blocks carry metadata for external tools)
- Not a schematic drawing tool (topology only; visual layout is downstream)

### Design Principles

1. **No ambiguity**: every syntactic construct has exactly one parse.
2. **No implicit behavior**: all connections, domains, and references are explicit.
3. **Fail loudly**: invalid input produces a precise, located diagnostic.
4. **AI-safe**: structure prevents generation of syntactically plausible but semantically broken designs.
5. **Minimal surface area**: every keyword earns its place.

## 2. Syntax Design

### Choice: Custom Block Syntax (Curly-Brace / Semicolon)

| Option | Rejected reason |
|--------|----------------|
| YAML | Significant whitespace causes generation errors; no block comments; indentation levels fragile |
| JSON | No comments; extremely verbose; no typed physical values; no procedural constructs |
| TOML | Flat tables only; no nested block structure needed for hierarchical modules |
| S-expressions | Unfamiliar to engineers; parenthesis matching fragile for AI generation |
| **Custom** | Explicit braces for scope; semicolons for statement termination; no significant whitespace; physical units as first-class literals; minimal keyword set |

The syntax is deterministic for recursive-descent parsing. No backtracking is needed.

### Minimal Complete Example

```hwdl
hwdl 1.0;

// === Part declarations (component library entries) ===

part "ATmega328P" {
    vendor: "Microchip";
    package: "DIP-28";
    datasheet: "https://example.com/atmega328p.pdf";
    pins {
        VCC:   power_in;
        GND:   power_gnd;
        AVCC:  power_in;
        AREF:  passive;
        RESET: input;
        PB0:   bidir;
        PB1:   bidir;
        TX:    output;
        RX:    input;
    }
}

part "GEN_CAP" {
    vendor: "generic";
    package: "0402";
    pins {
        P: passive;
        N: passive;
    }
}

// === Power domain declarations ===

power 3V3 {
    voltage: 3.3V;
    tolerance: 5%;
    max_current: 500mA;
}

power GND {
    voltage: 0V;
    type: "reference";
}

// === Module declaration ===

module MCU_Core {
    // Instance declarations
    U1: "ATmega328P";
    C1: "GEN_CAP" { value: 100nF; voltage_rating: 10V; }
    C2: "GEN_CAP" { value: 10uF;  voltage_rating: 10V; }

    // Explicit net declarations
    net uart_tx;
    net uart_rx;

    // Power connections
    connect U1.VCC  -> power.3V3;
    connect U1.GND  -> power.GND;
    connect U1.AVCC -> power.3V3;
    connect U1.AREF -> power.GND;
    connect C1.P    -> power.3V3;
    connect C1.N    -> power.GND;
    connect C2.P    -> power.3V3;
    connect C2.N    -> power.GND;

    // Signal connections
    connect U1.TX -> uart_tx;
    connect U1.RX -> uart_rx;

    // Electrical constraint metadata
    constraint {
        max_current:     200mA;
        operating_temp:  -40C to 85C;
        supply_voltage:  3.0V to 3.6V;
    }

    // Advisory placement hints (consumed by layout tool)
    placement {
        C1.near: "U1";
        C2.near: "U1";
        region:  "center";
    }

    // Advisory routing hints
    routing {
        power.3V3.width: 0.5mm;
        power.GND.width: 0.5mm;
        priority:        "power_first";
    }

    // Validation metadata (for ERC/DRC tooling)
    validate {
        erc:            true;
        drc:            true;
        footprint_check: true;
    }

    // Simulation metadata (for SPICE / other tools)
    simulate {
        model:     "spice";
        transient: "0s to 10ms";
        step:      1us;
    }
}

// === Compile target declaration ===

target kicad_v6 {
    format:     "kicad_v6";
    layers:     2;
    board_size: "50mm x 80mm";
}
```

## 3. Grammar

### 3.1 Notation

This grammar uses EBNF with these conventions:

- `::=` — production rule
- `|` — alternation
- `( ... )` — grouping
- `[ ... ]` — optional (zero or one)
- `{ ... }` — zero or more
- `"..."` — terminal literal
- `UPPER_CASE` — lexical token

### 3.2 File Structure

```ebnf
file            ::= version_decl top_decl* EOF

version_decl    ::= "hwdl" VERSION_LIT ";"
VERSION_LIT     ::= DIGIT+ "." DIGIT+
```

### 3.3 Top-Level Declarations

```ebnf
top_decl        ::= part_decl
                  | power_decl
                  | net_decl
                  | module_decl
                  | target_decl
```

### 3.4 Part Declaration

```ebnf
part_decl       ::= "part" STRING_LIT "{" part_item* "}"
part_item       ::= attr_stmt
                  | pin_block

pin_block       ::= "pins" "{" pin_stmt* "}"
pin_stmt        ::= IDENT ":" pin_dir ";"

pin_dir         ::= "power_in"
                  | "power_out"
                  | "power_gnd"
                  | "input"
                  | "output"
                  | "bidir"
                  | "passive"
                  | "open_drain"
                  | "no_connect"
```

### 3.5 Power Declaration

```ebnf
power_decl      ::= "power" IDENT "{" attr_stmt* "}"
```

### 3.6 Net Declaration

```ebnf
net_decl        ::= "net" IDENT ";"
```

### 3.7 Module Declaration

```ebnf
module_decl     ::= "module" IDENT "{" module_item* "}"

module_item     ::= instance_decl
                  | net_decl
                  | connect_stmt
                  | constraint_block
                  | placement_block
                  | routing_block
                  | validate_block
                  | simulate_block
```

### 3.8 Instance Declaration

```ebnf
instance_decl   ::= IDENT ":" STRING_LIT [ "{" attr_stmt* "}" ] ";"
```

The optional `{ attr_stmt* }` block overrides part-level attributes for this instance (e.g., component value, voltage rating). Instance `attr_stmt` keys must match attributes defined in the part declaration or be recognized generic attributes (`value`, `voltage_rating`, `tolerance`).

### 3.9 Connection Statement

```ebnf
connect_stmt    ::= "connect" endpoint "->" endpoint ";"

endpoint        ::= power_ref
                  | pin_ref
                  | net_ref

power_ref       ::= "power" "." IDENT
pin_ref         ::= IDENT "." IDENT
net_ref         ::= IDENT
```

**Disambiguation rule** (applied by the parser in order):
1. If the token sequence is `"power" "." IDENT` → `power_ref`
2. If the token sequence is `IDENT "." IDENT` → `pin_ref`
3. If the token is `IDENT` alone → `net_ref`

Restriction: The **left** endpoint of `->` must always be a `pin_ref`. The **right** endpoint may be `power_ref`, `net_ref`, or `pin_ref`. If right is `pin_ref`, an implicit net is created (valid only if both pins appear in no other connect statement assigning them to a different net).

### 3.10 Block Declarations

```ebnf
constraint_block ::= "constraint" "{" attr_stmt* "}"
placement_block  ::= "placement"  "{" attr_stmt* "}"
routing_block    ::= "routing"    "{" attr_stmt* "}"
validate_block   ::= "validate"   "{" attr_stmt* "}"
simulate_block   ::= "simulate"   "{" attr_stmt* "}"
```

Each block may appear at most once per module. The IR records all attribute key-value pairs verbatim for downstream consumption.

### 3.11 Target Declaration

```ebnf
target_decl     ::= "target" IDENT "{" attr_stmt* "}"
```

### 3.12 Attribute Statements

```ebnf
attr_stmt       ::= attr_key ":" attr_val ";"

attr_key        ::= IDENT [ "." IDENT ]

attr_val        ::= STRING_LIT
                  | range_val
                  | phys_val
                  | BOOL_LIT
                  | list_val
                  | IDENT

range_val       ::= phys_val "to" phys_val
phys_val        ::= num_lit UNIT
list_val        ::= "[" [ attr_val { "," attr_val } ] "]"
```

**`range_val` parse rule**: `phys_val "to" phys_val` is recognized only when the token after the first `phys_val` is the keyword `to`. If the `UNIT` of `low` and `high` differ, a type error is emitted.

### 3.13 Lexical Tokens

```ebnf
IDENT           ::= ALPHA { ALPHA | DIGIT | "_" }
                    (* must not be a reserved keyword *)

STRING_LIT      ::= '"' { CHAR } '"'
CHAR            ::= [^"\n\\] | "\\" ( '"' | "\\" | "n" | "t" )
BOOL_LIT        ::= "true" | "false"
num_lit         ::= [ "-" ] DIGIT+ [ "." DIGIT+ ]
                    [ ( "e" | "E" ) [ "+" | "-" ] DIGIT+ ]
ALPHA           ::= [A-Za-z_]
DIGIT           ::= [0-9]

UNIT            ::= "V"   | "mV"  | "uV"
                  | "A"   | "mA"  | "uA"  | "nA"
                  | "Ohm" | "kOhm"| "MOhm"
                  | "F"   | "uF"  | "nF"  | "pF"
                  | "H"   | "mH"  | "uH"  | "nH"
                  | "Hz"  | "kHz" | "MHz" | "GHz"
                  | "W"   | "mW"
                  | "mm"  | "cm"  | "um"  | "mil"
                  | "s"   | "ms"  | "us"  | "ns"  | "ps"
                  | "C"
                  | "%"

COMMENT         ::= "//" { [^\n] } "\n"?
WHITESPACE      ::= { " " | "\t" | "\r" | "\n" }+
(* COMMENT and WHITESPACE are discarded by the lexer *)
```

### 3.14 Reserved Keywords

The following identifiers are **reserved** and may not be used as `IDENT` in any context:

```
hwdl  part  power  module  net  connect  constraint
placement  routing  validate  simulate  target  pins
to  true  false
```

The following are reserved as **pin direction keywords** (valid only inside `pin_block`):

```
power_in  power_out  power_gnd  input  output  bidir
passive  open_drain  no_connect
```

> **Implementation note**: Pin direction keywords may be implemented as context-sensitive keywords (only reserved inside a `pin_block`) to allow their use as instance names elsewhere. However, forbidding them globally is simpler and recommended for v1.0.

## 4. AST Design

### 4.1 Source Location

Every AST node carries a `SourceLoc` for error reporting:

```python
@dataclass(frozen=True)
class SourceLoc:
    file: str    # absolute path to source file
    line: int    # 1-based line number
    col: int     # 1-based column number of first character
```

### 4.2 Node Types

```python
# ─── Root ────────────────────────────────────────────────────────────────────

@dataclass
class File:
    version:   "VersionDecl"
    top_decls: list["TopDecl"]  # PartDecl | PowerDecl | NetDecl | ModuleDecl | TargetDecl
    source:    str              # file path
    loc:       SourceLoc

@dataclass
class VersionDecl:
    major: int
    minor: int
    loc:   SourceLoc

# ─── Part ────────────────────────────────────────────────────────────────────

@dataclass
class PartDecl:
    name:  str            # from STRING_LIT (unquoted)
    attrs: list["AttrStmt"]
    pins:  list["PinStmt"]
    loc:   SourceLoc

@dataclass
class PinStmt:
    name:      str   # IDENT
    direction: str   # one of the 9 pin_dir values
    loc:       SourceLoc

# ─── Power ───────────────────────────────────────────────────────────────────

@dataclass
class PowerDecl:
    name:  str            # IDENT
    attrs: list["AttrStmt"]
    loc:   SourceLoc

# ─── Net ─────────────────────────────────────────────────────────────────────

@dataclass
class NetDecl:
    name: str   # IDENT
    loc:  SourceLoc

# ─── Module ──────────────────────────────────────────────────────────────────

@dataclass
class ModuleDecl:
    name:  str
    items: list["ModuleItem"]   # union of item types below
    loc:   SourceLoc

@dataclass
class InstanceDecl:
    ref_name:  str              # local name (U1, C1…)
    part_name: str              # part STRING_LIT (unquoted)
    overrides: list["AttrStmt"]
    loc:       SourceLoc

@dataclass
class ConnectStmt:
    src: "Endpoint"             # always PinRef in v1.0
    dst: "Endpoint"
    loc: SourceLoc

# Endpoint union
@dataclass
class PinRef:
    instance: str   # IDENT (left side of dot)
    pin:      str   # IDENT (right side of dot)
    loc:      SourceLoc

@dataclass
class PowerRef:
    domain: str   # IDENT after "power."
    loc:    SourceLoc

@dataclass
class NetRef:
    name: str   # IDENT
    loc:  SourceLoc

@dataclass
class ConstraintBlock:
    attrs: list["AttrStmt"]
    loc:   SourceLoc

@dataclass
class PlacementBlock:
    attrs: list["AttrStmt"]
    loc:   SourceLoc

@dataclass
class RoutingBlock:
    attrs: list["AttrStmt"]
    loc:   SourceLoc

@dataclass
class ValidateBlock:
    attrs: list["AttrStmt"]
    loc:   SourceLoc

@dataclass
class SimulateBlock:
    attrs: list["AttrStmt"]
    loc:   SourceLoc

# ─── Target ──────────────────────────────────────────────────────────────────

@dataclass
class TargetDecl:
    name:  str
    attrs: list["AttrStmt"]
    loc:   SourceLoc

# ─── Attributes ──────────────────────────────────────────────────────────────

@dataclass
class AttrStmt:
    key:   "AttrKey"
    value: "AttrVal"   # union below
    loc:   SourceLoc

@dataclass
class AttrKey:
    parts: list[str]   # 1 or 2 elements (dotted key)
    loc:   SourceLoc

# AttrVal union:
@dataclass
class StringVal:
    value: str
    loc:   SourceLoc

@dataclass
class PhysVal:
    number: float
    unit:   str
    loc:    SourceLoc

@dataclass
class RangeVal:
    low:  PhysVal
    high: PhysVal
    loc:  SourceLoc

@dataclass
class BoolVal:
    value: bool
    loc:   SourceLoc

@dataclass
class ListVal:
    items: list["AttrVal"]
    loc:   SourceLoc

@dataclass
class IdentVal:
    name: str     # bare IDENT used as value
    loc:  SourceLoc
```

### 4.3 Source Location Preservation Strategy

- The **lexer** attaches a `SourceLoc` to every token.
- The **parser** reads the `SourceLoc` from the **first token** consumed for each production and stores it on the resulting AST node.
- The `loc` field is **never derived or computed** — it always reflects the exact source position of the opening token of the construct.
- Multi-line constructs store only the start location; end location is omitted (can be added in v1.1 if diagnostics require spans).

## 5. Semantic Rules

### 5.1 Scope Rules

| Rule | Code | Description |
|------|------|-------------|
| SR-01 | E201 | Every `part` name (STRING_LIT) must be unique across the file. |
| SR-02 | E202 | Every `power` domain name must be unique across the file. |
| SR-03 | E203 | Every `module` name must be unique across the file. |
| SR-04 | E204 | Every `target` name must be unique across the file. |
| SR-05 | E205 | Within a module, every instance `ref_name` must be unique. |
| SR-06 | E206 | Within a module, every `net` name must be unique (and distinct from instance names). |
| SR-07 | E207 | A `net` name may not shadow a `power` domain name. |

### 5.2 Reference Rules

| Rule | Code | Description |
|------|------|-------------|
| RR-01 | E210 | Every `instance_decl` part name must reference a declared `part`. |
| RR-02 | E211 | Every `connect` `pin_ref` instance name must reference a declared instance in the same module. |
| RR-03 | E212 | Every `connect` `pin_ref` pin name must exist in the referenced instance's part `pins` block. |
| RR-04 | E213 | Every `connect` `power_ref` domain name must reference a declared `power` domain. |
| RR-05 | E214 | Every `connect` `net_ref` must reference a `net` declared in the same module scope (explicit net declarations are required; undeclared net names are errors). |
| RR-06 | E215 | Instance override attribute keys must match attributes declared in the target part, or be one of: `value`, `voltage_rating`, `tolerance`, `part_number`. |

### 5.3 Connectivity Rules

| Rule | Code | Description |
|------|------|-------------|
| CR-01 | E220 | A single pin may appear on the **left side** of at most one `connect` statement. Connecting the same pin twice is an error. |
| CR-02 | E221 | A single pin may belong to at most one net. If two connect statements would assign the same pin to different nets, that is an error. |
| CR-03 | E222 | A module must contain at least one `connect` statement (a module with no connections is meaningless and likely an error). This is a **warning** W010, not an error. |
| CR-04 | E223 | A `net` declared in a module but connected to zero pins is a **warning** W011. |

### 5.4 Electrical Rules (ERC)

These are checked when `validate { erc: true; }` is present; they are also **always** checked and produce warnings if ERC validation is not explicitly disabled.

| Rule | Code | Description |
|------|------|-------------|
| ER-01 | E300 | A pin with direction `power_in` must connect to a `power_ref`. Connecting to a signal net is an error. |
| ER-02 | E301 | A pin with direction `power_gnd` must connect to a `power_ref` whose `voltage` is `0V` or whose `type` is `"reference"`. |
| ER-03 | E302 | A pin with direction `power_out` may not connect to another `power_out` pin on the same net. |
| ER-04 | E303 | A pin with direction `output` may not connect to another `output` pin on the same net. |
| ER-05 | E304 | A pin with direction `no_connect` must not appear in any `connect` statement. |
| ER-06 | E305 | A net with **only** `input` pins (no driving pin) produces warning W020 "undriven net". |
| ER-07 | E306 | A `power_in` pin not connected to any net produces warning W021 "floating power pin". |

### 5.5 Unit Compatibility Rules

| Rule | Code | Description |
|------|------|-------------|
| UC-01 | E310 | In a `range_val`, `low.unit` and `high.unit` must be from the **same physical dimension** (e.g., both voltage, both temperature). Mixing units (e.g., `3.3V to 85C`) is an error. |
| UC-02 | E311 | `power.voltage` must have a voltage unit (`V`, `mV`, `uV`). |
| UC-03 | E312 | `power.max_current` must have a current unit (`A`, `mA`, `uA`, `nA`). |
| UC-04 | E313 | `power.tolerance` must have unit `%`. |
| UC-05 | E314 | Instance override `value` for capacitors must have capacitance unit; for resistors, resistance unit. (Capacitor/resistor detection is by convention: parts named per SR-01 with known pin directions. **Assumption**: part naming convention is not enforced by the grammar; UC-05 is a best-effort check emitting W030.) |

### 5.6 Structural Rules

| Rule | Code | Description |
|------|------|-------------|
| ST-01 | E400 | Each block type (`constraint`, `placement`, `routing`, `validate`, `simulate`) may appear **at most once** per module. |
| ST-02 | E401 | A `pins` block may appear at most once per `part`. |
| ST-03 | E402 | A `part` with no `pins` block is valid but produces warning W040 "part has no pins". |
| ST-04 | E403 | The `version_decl` must be the **first** non-comment token in the file. |
| ST-05 | E404 | `target` declarations may only appear at the **top level** (not inside modules). |

### 5.7 Error Examples

**E201 — Duplicate part name**
```hwdl
hwdl 1.0;
part "R" { pins { A: passive; B: passive; } }
part "R" { pins { A: passive; } }  // ERROR E201 line 3
```
Diagnostic: `E201 [3:1] duplicate part name "R"; first declared at line 2`

**E212 — Unknown pin**
```hwdl
connect U1.MISO -> spi_miso;
```
If `U1`'s part has no pin `MISO`:
Diagnostic: `E212 [5:10] pin "MISO" does not exist on part "ATmega328P" (instance "U1")`

**E300 — Power pin on signal net**
```hwdl
net my_signal;
connect U1.VCC -> my_signal;  // VCC is power_in
```
Diagnostic: `E300 [7:5] pin "U1.VCC" has direction "power_in" but is connected to signal net "my_signal"; expected a power domain`

**E310 — Range unit mismatch**
```hwdl
operating_range: 3.3V to 85C;
```
Diagnostic: `E310 [12:22] range bounds have incompatible units: "V" (voltage) vs "C" (temperature)`

## 6. Compiler Pipeline

### 6.1 Overview

```
Source Text
    │
    ▼
┌─────────┐
│  Lexer  │  → Token stream + SourceLoc on each token
└─────────┘
    │
    ▼
┌─────────────────────┐
│  Parser (recursive  │  → AST (File node)
│  descent)           │
└─────────────────────┘
    │
    ▼
┌──────────────────────────┐
│  Semantic Validator      │  → Annotated AST or error list
│  (scope + ref + ERC)     │
└──────────────────────────┘
    │
    ▼
┌──────────────────┐
│  Resolver        │  → Part refs resolved; pin refs validated;
│                  │     net membership computed
└──────────────────┘
    │
    ▼
┌──────────────────┐
│  Normalizer      │  → Physical values in base units;
│                  │     auto-net names assigned;
│                  │     net merge (transitive closure)
└──────────────────┘
    │
    ▼
┌──────────────────┐
│  IR Emitter      │  → hwdl_ir.json
└──────────────────┘
```

### 6.2 Lexer

**Input**: source string + file path  
**Output**: `list[Token]`

```python
@dataclass
class Token:
    kind:  str       # see token kinds below
    value: str       # raw text as it appeared in source
    loc:   SourceLoc
```

**Token kinds**:

| Kind | Pattern |
|------|---------|
| `KW_*` | Each reserved keyword (e.g., `KW_HWDL`, `KW_PART`) |
| `PINDIR_*` | Pin direction keywords |
| `IDENT` | `[A-Za-z_][A-Za-z0-9_]*` (not a keyword) |
| `STRING` | `"..."` (stored without quotes) |
| `VERSION` | `\d+\.\d+` |
| `NUMBER` | `[-]?\d+(\.\d+)?([eE][+-]?\d+)?` |
| `UNIT` | One of the unit strings (matched before IDENT) |
| `BOOL` | `true` / `false` |
| `ARROW` | `->` |
| `DOT` | `.` |
| `COLON` | `:` |
| `SEMICOLON` | `;` |
| `LBRACE` | `{` |
| `RBRACE` | `}` |
| `LBRACKET` | `[` |
| `RBRACKET` | `]` |
| `COMMA` | `,` |
| `EOF` | end of input |

**Lexer algorithm**:
1. Maintain `pos`, `line`, `col` cursors.
2. Skip whitespace and `//` comments (advance cursors accordingly).
3. At each position, attempt matches in this priority order:
   a. `//` → consume to end of line, loop
   b. `->` → ARROW
   c. Two-char sequences before single-char
   d. VERSION_LIT (digit+dot+digit) — only attempted after `hwdl` keyword context; **Assumption**: the lexer is context-free but VERSION_LIT is attempted greedily at `\d+\.\d+`; if it appears outside a version_decl, the parser emits E100.
   e. NUMBER: `-?\d+(\.\d+)?([eE][+-]?\d+)?` — attempt, but only if not followed by an IDENT char (to avoid lexing `3V` as `3` `V`).
   f. UNIT strings (longest match first, e.g., `kOhm` before `k`... units are matched only after a NUMBER token)
   g. Keywords (longest match first)
   h. IDENT
   i. STRING
   j. Punctuation

**Unit lexing rule**: After a NUMBER token, if the next characters match a UNIT string (from the ordered UNIT table, longest match first, no whitespace between), they are lexed as a UNIT token. This makes `100nF` lex as `NUMBER("100") UNIT("nF")`.

### 6.3 Parser

A **recursive descent parser** with one-token lookahead (LL(1) sufficient with the disambiguation rules in §3.9).

**Key parsing rules**:

- `parse_file()` → `File`
- `parse_top_decl()` → dispatches on current token kind
- `parse_part_decl()` → `PartDecl`
- `parse_module_decl()` → `ModuleDecl`
- `parse_module_item()` → dispatches on current keyword
- `parse_connect_stmt()` → `ConnectStmt`; calls `parse_endpoint()`
- `parse_endpoint()` → examines lookahead: `KW_POWER + DOT` → `PowerRef`; `IDENT + DOT` → `PinRef`; `IDENT` → `NetRef`
- `parse_attr_val()` → tries `range_val` (via tentative `phys_val` + `KW_TO`), then other forms
- `parse_phys_val()` → `NUMBER UNIT` (error if UNIT absent)

**Error recovery**: On a parse error, the parser emits a diagnostic and attempts to recover by consuming tokens until the next `;` or `}` (panic mode). This allows multiple errors to be reported in a single pass.

### 6.4 Semantic Validator

Runs over the AST in two sub-passes:

**Pass 1 — Symbol collection**:
- Build global symbol table: `parts`, `power_domains`, `modules`, `targets`
- Within each module: `instances`, `nets`
- Check uniqueness (SR-01 through SR-07)

**Pass 2 — Reference and ERC validation**:
- Resolve all `PinRef` instances and pins (RR-01, RR-02, RR-03)
- Resolve all `PowerRef` domains (RR-04)
- Resolve all `NetRef` names (RR-05)
- Check instance override keys (RR-06)
- Check connectivity rules (CR-01, CR-02)
- Check ERC rules (ER-01 through ER-07)
- Check unit compatibility (UC-01 through UC-05)
- Check structural rules (ST-01 through ST-05)

Both passes accumulate diagnostics in a `DiagList`. Compilation aborts (does not emit IR) if any `ERROR`-level diagnostics exist.

### 6.5 Resolver

**Input**: validated AST + symbol table  
**Output**: resolved AST with cross-references populated

Steps:
1. For each module, attach each `InstanceDecl` to its `PartDecl` node.
2. For each `ConnectStmt`, attach the `PinRef`'s `PartDecl.PinStmt` to the reference.
3. For each `PowerRef`, attach the `PowerDecl` node.
4. For each `NetRef`, attach the `NetDecl` node.

This produces a fully cross-referenced AST usable by the normalizer without further lookups.

### 6.6 Normalizer

**Net merger algorithm**:

1. Create a `net_registry`: `dict[net_name, list[PinEndpoint]]`
2. Initialize: for each `net_decl` in a module, create an empty entry.
3. For each `connect_stmt`:
   - Left is always `PinRef` → compute `pin_endpoint = (instance_name, pin_name)`
   - Right is `PowerRef` → use `"$power_{domain}"` as net name; add pin to that net
   - Right is `NetRef` → add pin to that net
   - Right is `PinRef` → generate auto-net name `"$auto_{index}"` (monotonic counter per module); add both pins to it
4. Check: if the same pin appears in two entries of `net_registry`, emit E221.
5. Compute transitive closure: if pin P is in net A via one connect, and P is also in net B via another (impossible after step 4 check), merge. (Step 4 prevents this; closure only needed if net aliasing is added in future versions.)

**Physical value normalization**:
- Convert all `PhysVal` to a canonical base unit for comparison:
  - Voltage → `V` (millivolt = ×0.001, microvolt = ×0.000001)
  - Current → `A`
  - Resistance → `Ohm`
  - Capacitance → `F`
  - etc.
- Store both original and normalized in IR.

**Auto-net naming**: sequential `$auto_0`, `$auto_1`, etc. per module. Names are prefixed with `$` to avoid collision with user net names (user identifiers cannot start with `$`).

### 6.7 IR Emitter

Serializes the normalized, resolved AST to JSON matching the schema in §7.

### 7.3 `attr_val` IR Encoding

All `attr_val` AST nodes serialize to a JSON object with a `kind` discriminator:

```json
{ "kind": "string",  "value": "Microchip" }
{ "kind": "phys",    "value": 3.3, "unit": "V", "normalized": 3.3, "base_unit": "V" }
{ "kind": "range",   "low": <phys>, "high": <phys> }
{ "kind": "bool",    "value": true }
{ "kind": "list",    "items": [ <attr_val>, ... ] }
{ "kind": "ident",   "value": "power_first" }
{ "kind": "integer", "value": 2 }
```

`INTEGER` is produced when a `num_lit` has no decimal point and no exponent and no UNIT — e.g., `layers: 2;`.

### 7.4 Net `kind` Values

| kind | Meaning |
|------|---------|
| `"signal"` | User-declared net (`net` keyword) |
| `"power"` | Net derived from a `power_ref` (name: `$power_{domain}`) |
| `"auto"` | Implicitly created by pin-to-pin connect (name: `$auto_{n}`) |

### 7.5 IR Guarantees for Consumers

A consumer receiving a valid IR (no errors in `diagnostics`) may rely on:

1. Every pin reference in `nets` resolves to an existing instance and pin.
2. Every instance references an existing part.
3. Every power net references an existing power domain.
4. No pin appears in more than one net.
5. Physical values are present in both original units and normalized base-unit form.
6. All `source_loc` fields are populated.

## 8. Error Model

### 8.1 Diagnostic Structure

```python
@dataclass
class Diagnostic:
    code:     str        # "E201", "W010", etc.
    severity: str        # "error" | "warning" | "info"
    message:  str        # human-readable description
    loc:      SourceLoc  # source location of the offending token
    context:  str | None # optional: e.g., "in module MCU_Core"
    hint:     str | None # optional fix suggestion
```

### 8.2 Severity Levels

| Severity | Meaning |
|----------|---------|
| `error` | Fatal; IR is not emitted; must be fixed |
| `warning` | Non-fatal; IR is emitted; should be reviewed |
| `info` | Informational; never blocks compilation |

### 8.3 Error Code Ranges

| Range | Category |
|-------|----------|
| E001–E099 | Lexer errors (unexpected character, unterminated string, invalid number) |
| E100–E199 | Parse errors (unexpected token, missing semicolon, mismatched braces) |
| E200–E299 | Scope & reference errors (undefined, duplicate) |
| E300–E399 | Type & unit errors |
| E400–E499 | Structural errors (duplicate blocks, wrong placement) |
| W001–W099 | Lexer warnings |
| W010–W099 | Semantic warnings (unconnected nets, no pins, etc.) |

## 9. Versioning Strategy

### 9.1 Version Declaration

Every HWDL file declares its language version as the first statement:

```hwdl
hwdl 1.0;
```

The compiler checks: `(file_major, file_minor) <= (compiler_major, compiler_minor)`.

### 9.2 Compatibility Policy

| Change type | Version bump | Policy |
|-------------|-------------|--------|
| Add new keyword | Minor (`1.0` → `1.1`) | Old files still parse; new keyword is optional |
| Add new block type | Minor | Old files compile; block is silently ignored if compiler version is lower |
| Add new pin direction | Minor | Older compilers emit W050 "unknown pin direction" |
| Add new built-in unit | Minor | Older compilers emit E004 |
| Remove keyword or construct | Major (`1.x` → `2.0`) | Old files may fail to compile against new compiler |
| Change semantic rule | Major | Must be clearly documented |
| Change IR schema | IR minor bump (separate) | IR version is independent of language version |

### 9.3 Compiler Version Registry

The compiler exports:
```python
HWDL_SUPPORTED_VERSIONS = [(1, 0)]  # list of (major, minor) tuples

def check_version(major: int, minor: int) -> bool:
    return (major, minor) in HWDL_SUPPORTED_VERSIONS
```

### 9.4 Forward-Compatibility Rule

If a file declares `hwdl 1.1` but the compiler only supports `hwdl 1.0`:
- Emit warning W001 `"file uses hwdl 1.1; compiler supports up to 1.0; unknown constructs will be skipped"`
- Continue parsing; emit errors only for constructs that cannot be parsed at all.

### 9.5 IR Versioning

The IR carries `"ir_version": "1.0"` independently. IR consumers should reject IR versions they do not recognize. The IR schema may be updated independently of the language version via a separate changelog.

### 10.3 Lexer Implementation

```python
import re
from dataclasses import dataclass
from typing import Iterator

# Build a single compiled regex with named groups, ordered by priority
TOKEN_PATTERNS = [
    ("COMMENT",   r'//[^\n]*'),
    ("WHITESPACE",r'[ \t\r\n]+'),
    ("VERSION",   r'\d+\.\d+'),
    ("ARROW",     r'->'),
    ("NUMBER",    r'-?\d+(?:\.\d+)?(?:[eE][+-]?\d+)?(?=[ \t\r\n;,\[\]{}])'),
    ("UNIT",      r'(?:kOhm|MOhm|Ohm|mV|uV|mA|uA|nA|uF|nF|pF|mH|uH|nH|kHz|MHz|GHz|ms|us|ns|ps|mil|mm|cm|um|mW|V|A|F|H|Hz|W|s|C|%)'),
    ("BOOL",      r'true|false'),
    ("STRING",    r'"(?:[^"\\]|\\.)*"'),
    ("KEYWORD",   r'hwdl|part|power|module|net|connect|constraint|placement|routing|validate|simulate|target|pins|to|power_in|power_out|power_gnd|input|output|bidir|passive|open_drain|no_connect'),
    ("IDENT",     r'[A-Za-z_][A-Za-z0-9_]*'),
    ("DOT",       r'\.'),
    ("COLON",     r':'),
    ("SEMICOLON", r';'),
    ("LBRACE",    r'\{'),
    ("RBRACE",    r'\}'),
    ("LBRACKET",  r'\['),
    ("RBRACKET",  r'\]'),
    ("COMMA",     r','),
    ("MISMATCH",  r'.'),   # catch-all for E001
]

MASTER_RE = re.compile(
    '|'.join(f'(?P<{name}>{pattern})' for name, pattern in TOKEN_PATTERNS)
)
```

**Important**: UNIT must be matched before IDENT (longer patterns first within UNIT). The regex alternation handles this via ordering.

**NUMBER lookahead** (`(?=[ \t\r\n;,\[\]{}])`) ensures `-3.3` is not greedily consumed when followed by a UNIT (`-3.3V` → NUMBER(`-3.3`) + UNIT(`V`)).

Actually, a simpler approach: lex NUMBER and UNIT as a combined pattern when together, but separate them. The safest implementation: after tokenizing a NUMBER, if the very next token (no whitespace) matches a UNIT, pop it off and emit both. A two-step approach avoids lookahead complexity.

### 10.4 Parser Implementation

```python
class Parser:
    def __init__(self, tokens: list[Token], source: str):
        self.tokens = tokens
        self.pos = 0
        self.source = source
        self.diags: list[Diagnostic] = []

    def peek(self) -> Token:
        return self.tokens[self.pos]

    def consume(self, expected_kind: str | None = None) -> Token:
        tok = self.tokens[self.pos]
        if expected_kind and tok.kind != expected_kind:
            self.error(E100, tok, f"expected {expected_kind}, got {tok.kind}")
        self.pos += 1
        return tok

    def match(self, kind: str) -> bool:
        return self.peek().kind == kind

    def expect(self, kind: str) -> Token:
        return self.consume(kind)

    # ... parse_file, parse_part_decl, parse_module_decl, etc.
```

Each `parse_*` method:
1. Consumes the opening token
2. Recursively parses sub-productions
3. Consumes the closing token / semicolon
4. Returns an AST node with `loc` set to the opening token's `SourceLoc`

### 10.7 Top-Level `compile()` Function

```python
def compile(
    source: str,
    source_path: str = "<stdin>",
    emit_on_warning: bool = True,
) -> tuple[dict | None, list[Diagnostic]]:
    """
    Returns (ir_dict, diagnostics).
    ir_dict is None if any ERROR-level diagnostic was produced.
    """
    tokens = lex(source, source_path)
    ast    = parse(tokens, source_path)
    diags  = validate(ast)
    if any(d.severity == "error" for d in diags):
        return None, diags
    resolved   = resolve(ast)
    normalized = normalize(resolved)
    ir         = emit_ir(normalized, source_path, diags)
    return ir, diags
```

### 11.4 Invalid Example 1 — Missing Version Declaration

```hwdl
part "GEN_RES" {
    pins { A: passive; B: passive; }
```

**Failure**: `E402 [1:1] version_decl must be the first statement in the file; found keyword "part"`

The parser's first action is to call `parse_version_decl()`. Finding `part` instead of `hwdl` is a hard parse error. No partial IR is emitted.

## Appendix A — Pin Direction Semantics

| Direction | Meaning | ERC Rules Applied |
|-----------|---------|------------------|
| `power_in` | Consumes power from a supply | Must connect to `power_ref`; floating = W021 |
| `power_out` | Provides a regulated supply | At most one per net (E302) |
| `power_gnd` | Ground reference | Must connect to 0V power domain (E301) |
| `input` | Unidirectional signal consumer | No driver check (W020) if net has no output |
| `output` | Unidirectional signal driver | At most one per net (E303) |
| `bidir` | Bidirectional signal | No driver conflict checked |
| `passive` | No defined direction (resistor, cap) | No ERC pin-direction rules |
| `open_drain` | Open-drain output (wired-AND capable) | Multiple allowed per net; no E303 |
| `no_connect` | Intentionally unconnected | Must not appear in `connect` (E304) |
